## 2. Shared vs. Duplicated

**Design tokens** — *Partially shared, structurally duplicated.* Canonical source is `packages/ui/src/tokens.css` (12-color dark-first palette + `[data-theme='light']`, `--space-1..8`, `--radius-*`, font-stack, focus ring). Web + desktop consume it correctly. **Mobile re-declares the same tokens by hand** in `apps/mobile/src/theme.ts:5-33` — colors+radii are byte-identical *today* but nothing enforces parity (no shared source, no parity test), so any `tokens.css` edit silently drifts mobile. **Explicit `tokens.css ↔ theme.ts` duplication:** all 12 colors copied as hex (`tokens.css:12-24` ↔ `theme.ts:5-18`); radii copied as numbers (`tokens.css:36-40` ↔ `theme.ts:28-33`); the spacing scale **diverges in shape** — CSS has 8 numbered steps, mobile 5 named ones (`xs/sm/md/lg/xl`) with no equivalent of `space-5/7/8`. Mobile has **no light palette** at all.

**Components** — *Shared within web+desktop, fully re-implemented on mobile.* `packages/ui` cleanly exports 21 typed components used by web/desktop. Mobile reimplements each as a divergent RN primitive: `AppButton` vs `Button` (no ghost/size/loading; `danger` is *outlined* on mobile but *solid-filled* on web), `StatusChip` vs `Badge`+`RunStatusBadge` (single-tint outline, no pulse, raw `status.replace(/_/g,' ')` vs curated labels), plus inline reimplementations of `ApprovalBar`, `MachineCard`, `WalletCard`, `EventTimeline`, `ScreenView`. `formatCents()` is duplicated with **divergent behavior** (`theme.ts:36-38` mishandles negatives → `$-0.05`; `CostPill.tsx:15-18` → `-$0.05`). *Within* web, the shared components are also bypassed (see §3). *Within* mobile, ApprovalBar / back-header / list-card / approval StyleSheets are copy-pasted across `RunDetailScreen` and `WorkflowRunsScreen`.

**Logos/assets** — *Largely duplicated/divergent (see §4).* The live mark is one theme-aware `currentColor` SVG (`Logo.tsx`), but the 6-stop horizon gradient is reimplemented **six times** and brand colors `#0b0e14`/`#e7eaf2` leak as standalone literals into `favicon.svg`, `gen-brand-assets.mjs`, mobile `components.tsx`, and `app.json`. `public/logo_light.svg` / `logo_dark.svg` are **orphaned** (referenced only in comments).

## 5. Responsive & Platform Notes

- **Breakpoints:** exactly **one `@media` query in the whole repo** — `(max-width: 800px)` in `apps/web/src/app.css:132-155` (collapses 220px sidebar → horizontal nav, `run-detail-grid` 2-col → 1-col). `packages/ui/src/styles.css` has **zero** media queries.
- **Coverage gaps:** binary flip only — **no tablet tier, no small-phone tier**; everything <800px gets one treatment, everything >800px is identical regardless of width. The 220px sidebar doesn't shrink before the 1100px cap, so a ~1280px window leaves asymmetric dead space. Several component grids never collapse (`oc-wallet-card__stats` is a hard `1fr 1fr`); some pages set query-unaware inline px (`LoginPage minWidth:300`).
- **Strengths:** intrinsic patterns exist — `.grid-cards` uses `auto-fill, minmax(260px,1fr)` (reflows with no query); `.app-main` is a centered fluid container capped at 1100px; `.page-header`/`.row` use `flex-wrap`.
- **Electron window reality:** `BrowserWindow` is `1280×840` with **no min size** (`main.ts:39`). This is a *resizable desktop window, not a mobile viewport* — at 1280px it sits in the web app's desktop branch (sidebar + 1100px cap, fills reasonably). But with no `minWidth`, a user can drag below 800px and get the **phone layout inside a desktop frame**; there's no desktop-specific intermediate tier (a 1280px window is treated the same as a 2560px monitor). Also: no `backgroundColor` set (white-flash risk before the dark SPA paints) and a native OS title bar unstyled against the dark palette.
- **React-Native unification boundary (central finding):** `@open-cowork/ui` is **DOM-only** — every component renders `div/span/svg/button/input` and ships behavior via a CSS file keyed off CSS custom properties. RN has no DOM, no `className`, no CSS custom properties, so **mobile cannot import the shared components** (stated in `theme.ts:1-4`, `App.tsx:6-8`). Unification therefore **cannot** mean "mobile imports the lib." What *can* be shared is the **design tokens** (today colors+radii are identical copies; only spacing-scale shape and `formatCents` actually diverge). The realistic seam: one platform-neutral token source generating both `tokens.css` and `theme.ts`, plus mirroring the *component contract* (variant names, sizes, tones, curated labels) in RN-native code rather than sharing implementation.

## 6. shadcn/ui Gap Summary

| Dimension | Current | shadcn convention | Gap |
|---|---|---|---|
| **Palette** | Dark-first (`color-scheme:dark`); saturated brand-blue accent `#5b8cff`; vivid status hues; flat `--color-*` list, duplicated in mobile TS | Light-first neutral baseline (zinc/slate), near-monochrome primary, saturated color reserved for destructive; role-paired semantic tokens (background/foreground/card/muted/ring) | Dark-first vs light-first; accent more brand-y than shadcn's near-neutral primary; flat color list (no `*-foreground` pairs); values duplicated across CSS + mobile TS with no shared source |
| **Whitespace / density** | Clean 4px scale used consistently, but tight: card padding `1rem`, no fixed control-height scale | Generous: card `p-6` (~1.5rem), fixed control heights `h-9`/`h-10`, roomy gaps | Reads "dashboard-dense" vs shadcn's airier default; no component-height scale so input/button rhythm can mismatch |
| **Borders vs shadows** | Borders-only; 1px `--color-border` everywhere; one inset logo ring; modal uses backdrop dim, no shadow | Border-forward **but** `shadow-sm` on cards/buttons and `shadow-lg` on overlays/dialogs | Closest dimension — but **total absence of elevation**: dialog reads flat (no `shadow-lg`); no `--shadow-*` token; "raised" faked purely via surface-color swap |
| **Radii** | Independent fixed px `4/8/12/999`; largest container 12px | Single `--radius` base (8px) with calc-derived sm/md/lg | Minor; values close, but global tuning needs editing 3 values not 1; 12px containers slightly rounder than shadcn's 8px |
| **Typography hierarchy** | **No type-scale tokens at all**; `.page-title` overloaded for h1+h2; section headings hand-set inline (`1rem`/`1.05rem`/`1.125rem`/`1.3rem` magic numbers) | One hierarchy via typography preset (text-xs..2xl, defined h1..h4, muted-foreground) applied uniformly | **Largest gap.** No single source for heading sizes/weights; same level looks different per page; blocks "one hierarchy everywhere" |
| **Variant consistency** | Only Button has a real variant API (`primary/secondary/danger/ghost` × `sm/md`, defaults secondary); naming differs per component (`variant` vs `tone`); no outline/link/icon; `danger` (not `destructive`) | Shared cva vocabulary: `default/secondary/destructive/outline/ghost/link` × `default/sm/lg/icon`, same words across Button/Badge/Alert | Missing outline/link/icon; `danger` vs `destructive`; no unified variant prop name; hand-written CSS instead of single cva source → manual cross-component parity |
| **Light-mode parity** | Full `[data-theme='light']` palette exists but wired only to a non-persisting manual toggle; no `prefers-color-scheme`; dark hardcoded default; mobile has no light theme | Light + dark first-class, light-default, respects OS, persisted (next-themes) | Values exist but behavior doesn't: no persistence, no OS detection, dark is hardcoded default, mobile dark-only — opposite of shadcn's light-first, OS-aware posture |

> **Already shadcn-aligned:** borders-over-shadows, the 4px spacing scale, a real `:focus-visible` ring token, and intrinsic `auto-fill` card reflow. Auditors agree the system is *close in spirit*; the work is conventions + a token/typography layer, not a rewrite.

## 7. Open Questions for Phase 2 (decide before implementing)

1. **Toolchain:** Adopt **Tailwind + shadcn/Radix** (net-new toolchain — there is *no* Tailwind, PostCSS, Radix, stylelint, or cva anywhere today; only a tiny `cx()` helper), or **keep the CSS-variables + `oc-*` system and apply shadcn *conventions*** (variant/size API, neutral-token discipline, single token source)? Note: Tailwind would be additive churn *and still wouldn't reach React Native*.
2. **Mobile unification depth:** Given the hard RN/DOM boundary, share **tokens only** (one neutral source generating `tokens.css` + `theme.ts`, plus a parity test), or *also* **mirror component APIs** (variant names/sizes/tones/curated labels/loading state) in RN-native code so the catalogs match 1:1 in contract while implementations stay platform-specific?
3. **Typography:** Since no type-scale token exists on either platform, what is the canonical scale (sizes + weights + heading levels), and how is it expressed for both CSS and RN? (This blocks "one hierarchy everywhere.")
4. **Palette posture:** Stay **dark-first** with a saturated brand-blue accent, or move toward shadcn's **light-first near-neutral** baseline? Decide light-mode wiring (OS detection + persistence) and whether mobile gains a light theme at all.
5. **Variant vocabulary:** Standardize on shadcn naming (`destructive` over `danger`, add `outline/link/icon`, unify `variant` vs `tone`)? This is a breaking rename across web + mobile if adopted.
6. **Brand / logo ambiguities needing user input:** (a) light/dark variant *rules* — the orphaned `logo_light.svg` (black mark) / `logo_dark.svg` (white mark) are named by *target background*, which is easy to misread as *mark color*; is the single `currentColor` `Logo.tsx` the intended source of truth, making both orphans deletable? (b) Is a **wordmark** needed (only a circular mark exists)? (c) clear-space and accent-color rules; (d) one web favicon format (SVG vs the separate Expo 64px PNG) and whether icon generation should move into CI rather than the manual script.

> **Where auditors agree:** the system is coherent and close to shadcn in spirit; the single biggest weakness is the **missing typography scale**, and the single biggest structural risk is **token duplication between `tokens.css` and `theme.ts`**. **Where data is thin / disputed:** "shadcn" is interpreted by every auditor as *conventions on the existing system* rather than a Tailwind+Radix adoption — that interpretation is an assumption, not a confirmed decision, hence Question 1.
